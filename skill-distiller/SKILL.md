# Skill Distiller（技能蒸馏器）

## TRIGGER when

- 用户说/问"帮我从这份文档生成一个 skill"
- 用户说/问"把这个分析文档蒸馏成技能"
- 用户说/问"这个文档的分析方法能做成模板吗"
- 用户提供了多个飞书文档链接，说"帮我提炼成一个可复用的分析 skill"
- 用户上传了包含分析结果的截图/图片，说"这个分析流程能做 skill 吗"

## DO NOT TRIGGER when

- 用户只是想读取飞书文档内容 → 用 feishu-toolchain 的文档读取能力
- 用户只是想问数据分析问题 → 用 intelligent-data-analysis 或 table-data-analysis
- 用户想创建"同事技能"（蒸馏人的行为）→ 用 colleague-skill

## Overview

**核心原则**：理解用户目的 → 反推思考过程 → 抽象分析方法论 → 生成可复用 Skill

**与旧版的核心区别**：

| 维度 | 旧版（0.0.1） | 新版（0.0.2） |
|-----|--------------|--------------|
| 蒸馏目标 | 具体问题 + 操作步骤 | 用户目的 + 思考方式 + 方法论 |
| TRIGGER | 写成具体问题 | 抽象为分析目的类型 |
| 分析步骤 | 写成操作流程 | 抽象为"遇到这类问题会怎么想" |
| 方法输出 | 列出具体方法名 | 说明什么场景用什么方法、为什么 |

**skill-distiller 的本质任务**：

把文档中的"具体问题实例"蒸馏成"用户的分析目的 + 遇到这类问题时的思考方式"：

```
原始文档：分析江苏6个城市公司的订单增幅，判断哪个城市好
    ↓ 蒸馏
Skill 输出：
  - 用户目的：比较多组实体在增幅维度上的表现，需要评估增长质量
  - 思考方式：1.基数效应 → 2.绝对贡献 → 3.统计显著性 → 4.综合判断
  - 方法选择：增幅对比时先考虑基数效应，再用增量贡献度衡量绝对价值
```

**这个 Skill 帮助你**：
1. **并行读取多个飞书文档**（文字+表格+图片截图）
2. **理解用户目的**：从具体问题中抽象出"想达成什么目的"
3. **反推思考过程**：用户遇到这类问题会怎么想、关注什么
4. **抽象分析方法论**：什么场景用什么方法、为什么（不是操作步骤）
5. **识别数据来源类型**：指标平台 vs 上传表格，明确区分
6. **生成 Skill 草稿**：输出到飞书文档

**输出信任度标注**：
- ✅ 明确：从文档中直接提取的内容
- 🔶 推断：基于上下文逻辑推断的内容
- ❓ 存疑：多个文档中不一致或置信度低的内容，需平台/人工校验

## 第一步：多源内容读取

### 1.1 读取飞书文档

**工具**：使用 feishu-toolchain 的 lark-cli 或 MCP 方式读取文档

支持读取的内容：
| 内容类型 | 处理方式 | 输出 |
|---------|---------|------|
| 文字段落 | 直接提取 | 原文 |
| 表格数据 | 提取行列结构 + 数据 | 结构化表格 |
| 列表/大纲 | 提取层次结构 | 带缩进的层级列表 |
| 嵌入图片 | 记录位置，后续用 vision 处理 | 图片位置标记 |
| 流程图/架构图 | 记录位置，后续用 vision 处理 | 图片位置标记 |

**并行读取**：多个文档链接同时读取，不分先后顺序

### 1.2 理解图片内容

对于文档中的截图、表格截图、流程图，使用 vision 能力理解：

| 图片类型 | 理解目标 |
|---------|---------|
| **表格截图** | 行列结构、数据内容、汇总行 |
| **流程图/架构图** | 步骤顺序、分支逻辑、循环关系 |
| **分析结果图** | 图表类型（折线/柱状/饼图）、数据趋势、关键结论 |
| **PPT/文档截图** | 页面布局、文字内容、章节结构 |

**输出格式**：
```
[图片N 图片描述]
- 内容类型：...
- 关键信息：...
- 置信度：🔶推断 / ✅明确
```

### 1.3 合并多源内容

合并规则：
- **文字优先**：同一内容，文字版优先于图片版
- **去重**：相同内容只保留一处，标注来源文档
- **冲突标注**：不同文档对同一问题的方法/结论不一致时，❓存疑并列出各文档的版本
- **来源追踪**：每个抽取内容标注来源文档（doc1/doc2/img1...）

## 第二步：识别数据来源类型（新增）

**重要性**：指标平台和上传表格的分析逻辑完全不同，必须区分。

### 2.1 数据来源类型

| 类型 | 特征 | 分析逻辑 |
|-----|------|---------|
| **指标平台** | 数据已经过处理，用户问"帮我分析这个指标" | 模型理解指标定义后直接分析，不需要解析原始数据 |
| **上传表格** | 用户上传原始数据，模型需要自己解析和处理 | 需要解析表格结构、识别指标/维度、处理格式问题 |
| **混合** | 部分数据在平台、部分上传 | 需分别处理后再整合 |
| **不明确** | 无法判断来源 | 🔶标注存疑，假设为上传表格（更通用） |

### 2.2 识别方法

从文档中识别数据来源：
- 文档提到"平台指标"、"报表系统"、"数据产品" → 指标平台
- 文档有原始数据表格、截图包含明细数据 → 上传表格
- 文档中提到具体数值（如"销售额100万"）但没说数据来源 → 存疑

### 2.3 输出格式

```markdown
### 数据来源识别

**识别结果**：✅指标平台 / ✅上传表格 / ✅混合 / 🔶存疑

**识别依据**：
- [从文档中提取的线索]

**分析逻辑说明**：
- 如果是指标平台：...
- 如果是上传表格：...
- 如果存疑：假设为上传表格处理，并说明需要确认
```

## 第三步：理解用户目的（核心！）

**这是 skill-distiller 最核心的步骤**。要从文档的具体问题中抽象出"用户真正想达成什么目的"，而不是写具体问题本身。

### 3.1 提取具体问题

首先明确文档中具体问的是什么（用于后续抽象对比，不要直接写入 Skill）：

```markdown
### 具体问题（原始）

**文档中的问题**：[原文摘录]
**涉及实体**：[如：6个城市公司]
**涉及指标**：[如：订单增幅]
**提问方式**：[如：哪个更好、为什么、趋势如何]
```

### 3.2 抽象用户目的

**核心原则**：抽象到"什么类型的目的"，而不是"什么具体问题"。

| 具体问题（❌不要这样写） | 抽象目的（✅应该这样写） |
|------------------------|------------------------|
| 江苏6个城市公司的订单增幅对比 | 比较多组实体在某一指标维度上的表现 |
| 分析无锡为什么增幅最高 | 识别异常表现并归因 |
| 本月销售额为什么下降 | 定位指标下降的原因 |
| 哪些城市表现好 | 评估多组实体的综合表现 |

**抽象方法**：
- 去掉具体实体（城市/门店/产品）
- 去掉具体指标（销售额/订单量）
- 保留分析模式（对比/归因/评估/探索）

### 3.3 推断深层意图

除了表层目的，还要推断用户真正想解决什么：

```markdown
### 深层意图推断

**表层目的**：比较多组实体的增幅表现

**深层意图**（🔶推断）：
- 想找出"表现最好"的实体，但不确定如何定义"好"
- 想给管理层一个客观的排名依据
- 想知道自己负责的区域在整体中处于什么位置

**置信度**：高/中/低（根据文档信息量判断）
```

### 3.4 用户目的输出格式

```markdown
## 用户目的（蒸馏核心）

**抽象目的类型**：
- 目的1：[抽象类型，如：多实体增幅对比]
- 目的2：[抽象类型，如：异常识别与归因]

**目的说明**：
[用1-2句话说明这个 Skill 解决什么类型的分析需求]

**适用场景**：
- 用户想比较多组实体在某个指标上的表现
- 用户想知道"谁更好"、"为什么"
- [其他适用场景]

**不适用场景**：
- [蒸馏出的 Skill 不适合的场景]
```

## 第四步：反推思考过程

从文档的分析步骤中，反推出"人遇到这类问题会怎么想"，而不是操作流程。

### 4.1 提取分析步骤

从文档中提取具体的分析步骤（用于后续抽象，不要直接写入 Skill）：

```markdown
### 具体分析步骤（文档中）

1. 计算各城市增幅 = (本期-上期)/上期
2. 按增幅排序
3. 结论：常州增幅最高（100%）
```

### 4.2 反推思考方式

**核心原则**：人遇到这类问题会怎么想，而不是机器要做什么操作。

| 具体步骤（❌不要这样写） | 抽象思考方式（✅应该这样写） |
|------------------------|--------------------------|
| 计算各城市增幅并排序 | 遇到增幅对比时，首先想到什么 |
| 判断哪个城市最好 | 如何定义"好"——增幅？增量？稳定性？ |
| 分析原因 | 异常表现如何归因 |

### 4.3 抽象思考过程模板

```markdown
## 思考过程（蒸馏结果）

**遇到这类问题时的思考顺序**：

1. **先想什么**： [如：先理解增幅的含义和局限性]
2. **再想什么**： [如：考虑基数效应（大小基数不能直接比）]
3. **然后想什么**： [如：衡量绝对贡献（增量大小）]
4. **最后想什么**： [如：综合判断（不能只看增幅%）]

**每个思考节点的关键问题**：
- 思考节点1： [人在这个节点会问自己什么问题]
- 思考节点2： [人在这个节点会问自己什么问题]

**思考依据**：
[这些思考顺序是从文档中哪些内容推断出来的]
```

### 4.4 思考过程示例

**原始文档**：分析江苏6个城市公司的订单增幅，判断哪个城市好

```markdown
### 思考过程（蒸馏结果）

**遇到增幅对比问题时的思考顺序**：

1. **先想基数效应**：增幅%受基数影响，小基数大幅增长可能统计意义不强
2. **再想绝对贡献**：不同基数的增幅不能直接比，要看绝对增量
3. **然后想统计显著性**：增幅是否在合理波动范围内
4. **最后综合判断**：不能只看增幅，要结合基数、增量、稳定性

**思考依据**：
- 文档中无锡(10万+50%)和常州(1万+100%)的对比，体现了对基数效应的考量
- 文档计算了绝对增量(5万 vs 1万)，体现了对绝对贡献的关注
```

## 第五步：抽象分析方法论

### 5.1 识别用户指定的方法

从文档中识别用户/分析师**明确要求**的方法：

```markdown
### 用户指定的方法（✅明确）

| 方法 | 文档中如何提及 | 来源 |
|-----|--------------|------|
| 环比增长率 | "增幅 = (本期-上期)/上期" | doc1 |
| 增量贡献度 | "绝对增量 = 基数 × 增幅" | doc1 |
```

### 5.2 推断方法选择逻辑

**核心**：不是列出方法名，而是说清楚"什么场景用什么方法、为什么"。

```markdown
### 方法选择逻辑

**遇到 [用户目的类型] 时**：

| 场景 | 选用方法 | 选择原因（为什么） | 统计依据 |
|-----|---------|------------------|---------|
| 比较增幅快慢 | 环比增长率 | 衡量增速的标准化方法 | (本期-上期)/上期 |
| 不同基数增幅对比 | 增量贡献度 | 规避基数效应，衡量绝对贡献 | 增量/总增量 |
| 判断增幅是否异常 | 增幅偏离度 | 识别统计显著异常 | (增幅-均值)/σ |
| [其他场景] | [方法] | [为什么] | [统计依据] |

**不能直接比较的情况**：
- 基数差异大时，不能直接比增幅%
- 需要分层或分类后再对比
```

### 5.3 方法使用优先级

**重要**：用户明确指定的方法必须优先执行，模型只在用户没指定时自判断。

```markdown
### 方法使用优先级

**用户指定了方法时**：
→ 必须按用户指定的方法执行
→ 用户可能说"帮我用XX方法分析"

**用户没指定方法时**：
→ 模型根据目的类型自主选择方法
→ 选择逻辑：见上方"方法选择逻辑"
→ 选择后需说明"为什么选这个方法"
```

### 5.4 用户没有指代方法时模型自判断

```markdown
### 方法自判断规则

当用户没有明确指定方法时，模型根据以下规则判断：

**判断规则**：
1. 先识别用户的分析目的类型（增幅对比？归因？趋势？）
2. 再看数据特征（有没有基数差异？有没有时间序列？）
3. 根据目的+数据特征选择最合适的方法
4. 选择后输出："选用XX方法，因为..."

**常见场景的方法选择**：
| 场景 | 推荐方法 | 选择原因 |
|-----|---------|---------|
| 多实体增幅对比，有基数差异 | 增量贡献度 + 增幅偏离度 | 先规避基数效应，再判断统计显著性 |
| 指标下降归因 | 贡献度拆解 + 维度下钻 | 先拆解各维度贡献，再定位原因 |
| 趋势分析 | 环比序列 + 趋势拟合 | 先看短期波动，再看长期趋势 |
```

## 第七步：Schema 补全

### 7.1 推断 TRIGGER when

**核心原则**：TRIGGER 写的是"什么类型的分析目的"，**不是**"什么具体问题"。

```markdown
## TRIGGER when

- 当用户想[抽象目的类型]时
- 当用户问"[意图类型]的问题时"
- 当用户上传[数据来源类型]并要求[目的]时

（注意：不要写具体实体、具体指标、具体数值）
```

**示例对比**：

| ❌ 错误（太具体） | ✅ 正确（抽象目的） |
|-----------------|-------------------|
| 用户上传江苏6个城市公司的订单增幅数据 | 用户想比较多组实体在某一指标上的表现 |
| 用户问"哪个城市表现最好" | 用户想评估多组实体的综合表现，需要判断优劣 |

### 7.2 推断 DO NOT TRIGGER when

```markdown
## DO NOT TRIGGER when

- 用户上传了 Excel/CSV 文件要求分析 → 用 table-data-analysis-pe
- 用户只是想查指标数据 → 直接查询平台，不触发分析
- 用户问的是 [其他分析类型] → 用对应的其他 Skill
```

### 7.3 推断 Red Flags

基于文档中暴露的问题和通用分析陷阱推断：

```markdown
## Red Flags - 停下来检查

当有以下想法时，停下来重新审视：

- "增幅大的就是好" → **必须考虑基数效应，分析绝对增量**
- "直接输出结论" → **必须先说明分析方法论，给出统计学推断**
- "用户没指定方法，随便用" → **用户没指定时，模型必须自己判断方法并说明原因**
- [其他检查项]
```

### 7.4 推断统计陷阱

```markdown
## 统计陷阱警示

| 陷阱 | 说明 |
|-----|------|
| 基数效应 | 小基数大幅增长可能统计意义不强，不能直接比增幅% |
| 辛普森悖论 | 分组与整体趋势可能相反，需同时看分组和整体 |
| 相关性≠因果性 | 不能因为相关就断定因果 |
| [其他陷阱] | [说明] |
```

## 常见错误

| 错误 | 正确做法 |
|-----|---------|
| **目的写成具体问题** | TRIGGER 必须抽象到"目的类型"，不能写具体问题 |
| **步骤写成操作流程** | 分析方法论要写"人怎么想"，不是"机器怎么做" |
| **不区分数据来源** | 必须明确是指标平台还是上传表格，分析逻辑不同 |
| **用户指令不优先** | 用户明确指定方法时必须优先执行，没指定才自判断 |
| **方法自判断无依据** | 自判断时要说明"为什么选这个方法" |
| **方法误判** | 区分"明确提及"和"推断"，不把推断当明确 |
| **Schema 过度补全** | 推断的部分明确标注🔶推断，不假装是明确的 |

## Red Flags - 停下来检查

当你有以下想法时，停下来重新审视：

- "TRIGGER 写成具体问题" → **TRIGGER 必须抽象到目的类型，去掉具体实体/指标/数值**
- "分析步骤写成操作流程" → **要写"人遇到这类问题会怎么想"，不是"读取表格→检测格式"**
- "不区分数据来源" → **必须明确：指标平台 vs 上传表格，分析逻辑完全不同**
- "直接输出结论" → **必须先说明分析方法论，给出统计学推断过程**
- "用户没指定方法，随便选" → **必须先识别用户目的类型，再根据目的选择方法，并说明原因**
- "增幅大的就是好" → **必须考虑基数效应，分析绝对增量和统计显著性**
