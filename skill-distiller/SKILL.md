# Skill Distiller（技能蒸馏器）

## TRIGGER when

- 用户说/问"帮我从这份文档生成一个 skill"
- 用户说/问"把这个分析文档蒸馏成技能"
- 用户说/问"这个文档的分析方法能做成模板吗"
- 用户提供了多个飞书文档链接，说"帮我提炼成一个可复用的分析 skill"
- 用户上传了包含分析结果的截图/图片，说"这个分析流程能做 skill 吗"

## DO NOT TRIGGER when

- 用户只是想读取飞书文档内容 → 用 feishu-toolchain 的文档读取能力
- 用户只是想问数据分析问题 → 用 intelligent-data-analysis 或 table-data-analysis
- 用户想创建"同事技能"（蒸馏人的行为）→ 用 colleague-skill

## Overview

**核心原则**：多源理解 → 方法反推 → Schema 补全 → 结构化输出

这个 Skill 帮助你：
1. **并行读取多个飞书文档**（文字内容、表格数据、图片截图）
2. **理解分析目的和类型**（归因/趋势/探索/评估/综合）
3. **反推分析方法**（从分析步骤中识别用了什么方法，明确 vs 推断）
4. **抽取呈现格式**（结论结构、建议格式、数据链路）
5. **补全 Skill Schema**（TRIGGER、DO NOT TRIGGER、Red Flags 等）
6. **生成 Skill 草稿**（输出到飞书文档或本地文件）

**输出信任度标注**：
- ✅ 明确：从文档中直接提取的内容
- 🔶 推断：基于上下文逻辑推断的内容
- ❓ 存疑：多个文档中不一致或置信度低的内容，需平台/人工校验

## 第一步：多源内容读取

### 1.1 读取飞书文档

**工具**：使用 feishu-toolchain 的 lark-cli 或 MCP 方式读取文档

支持读取的内容：
| 内容类型 | 处理方式 | 输出 |
|---------|---------|------|
| 文字段落 | 直接提取 | 原文 |
| 表格数据 | 提取行列结构 + 数据 | 结构化表格 |
| 列表/大纲 | 提取层次结构 | 带缩进的层级列表 |
| 嵌入图片 | 记录位置，后续用 vision 处理 | 图片位置标记 |
| 流程图/架构图 | 记录位置，后续用 vision 处理 | 图片位置标记 |

**并行读取**：多个文档链接同时读取，不分先后顺序

### 1.2 理解图片内容

对于文档中的截图、表格截图、流程图，使用 vision 能力理解：

| 图片类型 | 理解目标 |
|---------|---------|
| **表格截图** | 行列结构、数据内容、汇总行 |
| **流程图/架构图** | 步骤顺序、分支逻辑、循环关系 |
| **分析结果图** | 图表类型（折线/柱状/饼图）、数据趋势、关键结论 |
| **PPT/文档截图** | 页面布局、文字内容、章节结构 |

**输出格式**：
```
[图片N 图片描述]
- 内容类型：...
- 关键信息：...
- 置信度：🔶推断 / ✅明确
```

### 1.3 合并多源内容

合并规则：
- **文字优先**：同一内容，文字版优先于图片版
- **去重**：相同内容只保留一处，标注来源文档
- **冲突标注**：不同文档对同一问题的方法/结论不一致时，❓存疑并列出各文档的版本
- **来源追踪**：每个抽取内容标注来源文档（doc1/doc2/img1...）

## 第二步：内容理解

### 2.1 识别文档类型

| 类型 | 特征 | 示例 |
|-----|------|------|
| **分析报告** | 有数据、有结论、有建议 | "XX 指标分析报告" |
| **分析方案** | 有方法、有框架、有步骤 | "XX 问题分析方案" |
| **研究综述** | 有调研、有对比、有结论 | "竞品分析/技术调研" |
| **综合文档** | 包含多种内容类型的混合文档 | 混合型 |

### 2.2 识别分析目的

根据内容判断分析的核心目的：

| 目的 | 关键词 | 典型内容 |
|-----|-------|---------|
| **归因分析** | 原因、因素、贡献、拆解 | 为什么指标变化了 |
| **趋势分析** | 趋势、变化、走势、环比同比 | 指标如何变化 |
| **探索分析** | 规律、发现、洞察、异常 | 找规律、发现异常 |
| **评估分析** | 效果、评估、对比、优劣 | 评估方案/活动效果 |
| **预测分析** | 预测、预估、未来、走势 | 预测未来趋势 |

### 2.3 结合用户补充描述

用户补充的描述用于确认：
- 分析的具体场景（哪个业务线、哪个指标）
- 分析的目标受众（给谁看、解决什么问题）
- 特殊的约束条件（时间范围、口径要求等）

**未提供时**：基于文档内容推断，并在草稿中标注❓存疑

## 第三步：方法反推

这是最关键的步骤。需要从文档的分析步骤中反推使用了什么方法。

### 3.1 抽取明确方法

直接从文档中识别到的方法（✅明确）：

| 方法 | 识别特征 | 文档中的典型表述 |
|-----|---------|----------------|
| 指标归因 | 拆解子指标、计算贡献度 | "拆解为XX子指标"、"贡献度=..." |
| 维度归因 | 按维度分组、计算各维度贡献 | "按地区/渠道/门店拆分" |
| 趋势对比 | 同比/环比、时间序列 | "环比增长XX"、"同比下降XX" |
| 同期群分析 | 按时间分组、对比留存 | "Cohort分析"、"第N日留存" |
| 漏斗分析 | 步骤转化、流失率 | "转化率=.../..." |
| A/B测试 | 实验组/对照组、差异显著性 | "A组/B组差异"、"p值" |
| 异常检测 | 异常值、偏离均值 | "偏离均值XX%"、"异常值" |
| 相关分析 | 相关系数、相关关系 | "与XX相关性为..." |

### 3.2 推断隐含方法

文档中没有直接写，但可以从分析步骤中推断的方法（🔶推断）：

推断策略：

| 推断线索 | 可能的隐含方法 |
|---------|--------------|
| 有子指标数据但没写归因 | 指标归因 |
| 有维度拆分但没写维度归因 | 维度归因 |
| 有 Top N 排名 | 贡献计算/排序分析 |
| 有下钻（从汇总到明细） | 智能下钻 |
| 有分组对比但没写统计检验 | 对比分析（可能缺统计检验） |
| 只给结论没给数据支撑 | 逻辑推断（可能缺数据验证）|

**标注格式**：
```
🔶推断：{方法名}
  依据：{从什么内容推断而来}
  置信度：{高/中/低}
```

### 3.3 合并方法列表

输出格式：
```
## 识别到的方法

### ✅ 明确方法
1. {方法名}：{文档中如何提及}
2. ...

### 🔶 推断方法
1. {方法名}（🔶推断，高置信度）
   依据：{...}
2. {方法名}（🔶推断，中置信度）
   依据：{...}

### ❓ 存疑
1. {内容}：{哪些文档说法不一}
```

## 第四步：格式抽取

### 4.1 抽取输出结构

从文档中抽取结论和建议的呈现结构：

| 维度 | 抽取内容 |
|-----|---------|
| **章节层次** | 文档有几层标题，各层次的关系 |
| **结论位置** | 结论前置还是娓娓道来 |
| **数据展示** | 用表格/图表/文字描述，数据格式 |
| **建议呈现** | 建议是否有优先级，如何组织（列表/表格/段落） |

### 4.2 抽取结论呈现

```
结论格式（从文档中抽取）：
- 层级：{结论有几层（如：总-分）}
- 数据支撑：{有/无/部分} 明确数据
- 建议格式：{表格/列表/段落/无建议}
- 优先级：{有/无 优先级标注}
```

### 4.3 抽取注意事项

从文档中识别到的"坑"或"警示"：

| 类型 | 示例 |
|-----|------|
| **口径说明** | "XX 指标口径为..." |
| **数据限制** | "因数据口径不一致，仅对比XX期间" |
| **方法局限** | "本期未做 AB 测试，仅作描述性分析" |
| **异常情况** | "XX 日期因系统故障，数据缺失" |

## 第五步：Schema 补全

根据前几步的结果，补全 Skill 的各个必需部分。

### 5.1 推断 TRIGGER when

基于分析目的和使用场景推断：

```
## TRIGGER when
- 用户问/说"分析{分析目的关键词}"
- 用户问/说"看看{指标/现象}的原因"
- 用户问/说"帮我分析{业务场景}"
- 需要对{分析类型}进行分析时
```

### 5.2 推断 DO NOT TRIGGER when

基于分析范围和已有 Skill 区分：

```
## DO NOT TRIGGER when
- 用户上传了 Excel/CSV 文件要求分析 → 用 table-data-analysis
- 用户问的是{其他分析类型} → 用对应的其他 Skill
- 用户只是想查指标数据 → 直接查询平台，不触发分析
```

### 5.3 推断 Red Flags

基于文档中暴露的问题和通用分析陷阱推断：

```
## Red Flags - 停下来检查
- 当{出现某种情况}时，停下来重新审视
- 当{出现另一种情况}时，检查{检查项}
- ...
```

### 5.4 推断统计陷阱（如适用）

根据分析类型推断可能涉及的统计陷阱：

| 分析类型 | 可能的统计陷阱 |
|---------|--------------|
| 趋势分析 | 数据泄露（用未来数据预测过去） |
| 对比分析 | 辛普森悖论（分组与整体趋势相反） |
| 归因分析 | 相关性≠因果性 |
| 小样本分析 | 基数效应（小维度波动大） |
| 用户分析 | 幸存者偏差（只分析"存活"用户）|

- 章节层次：{N} 层

**用户指定位置**：按用户指定的路径或飞书文档写入

1. 完整的 Skill 草稿（Markdown 格式）
2. 置信度标注汇总（便于平台测试时重点关注）

## Evolution 模式：追加文档

当用户提供更多文档时：

1. 读取新增文档（按第一步方式）
2. 与已有草稿对比：
   - 新增方法 → 补充到方法列表
   - 冲突方法 → ❓存疑，列出各版本
   - 新增格式 → 补充到输出格式
3. 重新生成更新后的草稿
4. 更新置信度标注

## 常见错误

| 错误 | 正确做法 |
|-----|---------|
| **方法误判** | 区分"明确提及"和"推断"，不把推断当明确 |
| **格式幻觉** | 只抽取文档中实际呈现的格式，不创造新格式 |
| **Schema 过度补全** | 推断的部分明确标注🔶推断，不假装是明确的 |
| **多文档冲突不处理** | 冲突内容❓存疑，列出各版本，不强行统一 |
| **忽略图片内容** | 截图中的表格和流程图同等重要，必须用 vision 理解 |
| **跳过用户补充描述** | 用户描述是确认分析目的的关键输入，必须结合 |

## Red Flags - 停下来检查

当你有以下想法时，停下来重新审视：

- "这个方法应该是归因" → 检查文档是否有明确提及，优先从子指标/维度等特征判断
- "文档结构可以这样组织" → 检查是否文档中实际是这样呈现的，不创造新结构
- "TRIGGER 应该是..." → TRIGGER 需基于实际使用场景推断，不能凭空编造
- "所有推断方法都加进去" → 优先明确方法，推断方法标注置信度
